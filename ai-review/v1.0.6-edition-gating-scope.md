# v1.0.6 scope — edition detection (#134) + over-gating (#136)

**Status:** SCOPING (analysis complete; awaiting friend's Gold fixtures to confirm before implementation)
**Date:** 2026-05-28
**Author:** code-grounded analysis during the v1.0.5 wait window

These two issues are the same root knot and should ship together in v1.0.6.

---

## Root cause (one knot, two symptoms)

### The mechanism mismatch

The edition probe and the actual reports use **different XML mechanisms**:

- **Edition probe** (`apps/mcp-server/src/capability.ts` → `LEGACY_TB_PROBE`): sends a
  legacy **Report form** request (`TALLYREQUEST=Export Data, TYPE=Data, ID=Trial Balance`).
  TallyPrime Silver answers this with `STATUS=0` / empty `<DATA/>`. Some **networked Gold**
  setups also return a shape that doesn't match the `<TBROW|<DSPACCNAME|<LEDGER` regex, OR
  the 5s probe times out behind the gateway. → **false-Silver (#134).**

- **Actual reports** (`packages/tdl-engine/templates/*.xml` via `runTdlReport`): use inline
  **TDL Collection** projections. Verified collection types:

  | TDL template | `<COLLECTION><TYPE>` | Master-safe (works on Silver)? |
  |---|---|---|
  | `trial-balance.xml` | **Ledger** | ✅ yes |
  | `profit-loss.xml` | **Group** | ✅ yes |
  | `balance-sheet.xml` | **Group** | ✅ yes |
  | `day-book.xml` | Voucher | ❌ Gold-only |
  | `sales-register.xml` | Voucher | ❌ Gold-only |

  These project `$OpeningBalance` / `$DebitTotals` / `$CreditTotals` / `$ClosingBalance` —
  pure master-level fields, **no voucher walk.**

**So the probe tests a mechanism (legacy Report form) that NONE of the real reports use.**
That's the #134 design flaw: a false-Silver verdict from the probe doesn't reflect whether
the master-safe TDL reports actually work (they do, on every edition).

### The over-gating (#136)

`apps/mcp-server/src/server.ts` has a single capability flag `voucherQueriesViable` and a
single `gateOnVouchers()` guard applied to 5 tools:

| Tool | server.ts | Underlying mechanism | Correctly gated? |
|---|---|---|---|
| `tally_export_vouchers` | :252 | Voucher collection | ✅ yes — genuinely needs Gold |
| `tally_get_ledger_closing_balance` | :280 | **TDL trial-balance (Ledger)** | ❌ **OVER-GATED** — master-safe |
| `tally_get_group_closing_balances` | :316 | **TDL trial-balance (Ledger)** | ❌ **OVER-GATED** — master-safe |
| `tally_run_audit_lite` | :348 | Mixed (TB/P&L/BS + voucher checks) | ⚠ partial |
| `tally_export_dashboard` | :373 | Mixed (TB/P&L/BS + vouchers) | ⚠ partial |

The closing-balance connector (`packages/report-engine/src/connectors/ledger-balance.ts`)
documents its own Silver-safety in lines 10-24: *"Backed by the inline-TDL trial-balance
template… Earlier implementations used a raw Collection+FETCH `$ClosingBalance` projection
[that] wedges the gateway on Silver… Switching to the TDL template removes the wedge."*
Yet the tools are gated as if they need vouchers. They don't.

Note: `tally_read_report` for trial-balance/profit-loss/balance-sheet is **NOT** gated and
already routes through the TDL templates — so those reports already work on Silver today.
The over-gating only bites the two closing-balance tools (fully) and audit-lite/dashboard
(partially).

## The fix (v1.0.6)

### Part A — split the capability model (`capability.ts`)

Replace the single `voucherQueriesViable` with two flags:

```ts
export interface TallyCapabilities {
  reachable: boolean;
  edition: "silver" | "gold" | "unknown";
  masterQueriesViable: boolean;   // NEW — Ledger/Group collections (TB, P&L, BS, closing balances)
  voucherQueriesViable: boolean;  // vouchers, day-book, sales-register, full audit-lite
  detectedAt: string;
  message: string;
}
```

- `masterQueriesViable` is true whenever Tally is reachable and serves a Ledger/Group
  collection (essentially always — Silver included).
- `voucherQueriesViable` stays Gold-only.

### Part B — fix the probe (#134)

Stop probing the legacy Report form. Probe what the reports actually use:

1. **Master probe**: a cheap TDL Ledger collection (the same `trial-balance` template path,
   or a 1-row Ledger collection). If it returns ledger rows → `masterQueriesViable = true`.
   This is what determines whether TB/P&L/BS/closing-balances work — and it's the same
   mechanism, so no more false negatives.
2. **Voucher probe**: a tightly-bounded Voucher collection (e.g. 1-day window) with a short
   deadline. Success → `voucherQueriesViable = true` (Gold). Timeout/empty → false (Silver).
3. `edition` becomes a derived label: `gold` if voucher-viable, else `silver` if
   master-viable, else `unknown`.

Keep `config.tally.assumedEdition` + `unsafeSlow` overrides as-is (they already bypass the
probe — the friend's safety-net relies on this).

### Part C — re-gate tools (`server.ts`)

| Tool | New gate |
|---|---|
| `tally_get_ledger_closing_balance` | `gateOnMasters` (NEW — passes on Silver) |
| `tally_get_group_closing_balances` | `gateOnMasters` |
| `tally_export_vouchers` | `gateOnVouchers` (unchanged) |
| `tally_run_audit_lite` | `gateOnVouchers` for now (the voucher checks need Gold). v1.1.0: split audit into master-only + voucher checks so Silver gets a partial score. |
| `tally_export_dashboard` | `gateOnVouchers` for now (pulls vouchers). v1.1.0: master-only dashboard variant. |

`gateOnMasters(ctx)` returns the fail-fast error only when `!masterQueriesViable` (i.e. Tally
unreachable) — so on any reachable edition, closing-balance tools run.

### Part D — message clarity

`tally_get_capabilities` message should distinguish: "Masters (TB/P&L/BS/balances) ✅;
Vouchers (day-book/audit-lite) ❌ on this edition — use file-import for vouchers."
