**Tag:** `v1.0.0` currently points at `db60fc7` (the buggy commit)
**Release object:** none exists (`gh release view v1.0.0` → "release not found")

## The first failure (Run #26491196656)

Triggered by `git push origin v1.0.0`. Failed at the **Build signed installer** step:

```
SignTool Error: The specified PFX password is not correct.
```

**Root cause:** the `CSC_KEY_PASSWORD` GitHub Secret didn't match the password used when exporting the `.pfx` in Step 1 of `docs/v1-installer-phase4-secrets-setup.md`. Likely a typo when setting the secret.

**Fix:** Re-set `CSC_KEY_PASSWORD` via `gh secret set CSC_KEY_PASSWORD` (interactive, password never enters chat). Verified by `gh secret list` showing both secrets timestamped fresh.

## The second failure (Run #26491457966, workflow_dispatch on v1.0.0)

Re-triggered via `gh workflow run release.yml --ref v1.0.0 -f tag=v1.0.0` after the password fix. Signing **succeeded** on all 5 binaries (node.exe, TallyMCP.exe, elevate.exe, uninstaller, TallyMCP-Setup-v1.0.0.exe). Then failed at the same overall step with a NEW error:

```
× Cannot cleanup:
  Error: GitHub Personal Access Token is not set, neither programmatically,
  nor using env "GH_TOKEN"
    at PublishManager.artifactCreatedWithoutExplicitPublishConfig
```

**Root cause:** electron-builder detected the v1.0.0 tag context + saw `publish: github` in `apps/configurator/electron-builder.yml` (Task 8) and tried to **auto-publish to GitHub Releases** via its built-in publisher — which needs a `GH_TOKEN` env var the workflow doesn't set.

The workflow design (Task 9) was: electron-builder builds + signs, then `softprops/action-gh-release@v2` (a separate workflow step) does the upload. The `publish: github` config in electron-builder.yml is needed so electron-builder generates `latest.yml` (for electron-updater consumption), but it shouldn't try to upload artifacts itself.

## The fix (commit `5506a82` on `main`)

`package.json` root script:

```diff
- "package": "... && electron-builder --config electron-builder.yml --win && ..."
+ "package": "... && electron-builder --config electron-builder.yml --win --publish never && ..."
```

The `--publish never` flag tells electron-builder to skip its auto-publish step even when the `publish:` provider config is present. The config stays in `electron-builder.yml` because we still need it for `latest.yml` generation (electron-updater feed) — only the **upload phase** is skipped.

**Verification:** local `pnpm package` invocation parses the new flag cleanly. CI doesn't yet have this fix on the v1.0.0 commit.

## The decision: how to ship v1.0.0

The v1.0.0 tag on origin currently points at `db60fc7` (version-bump commit, before the `--publish never` fix). The fix is on main at `5506a82`. To re-fire the release workflow against a commit that includes the fix, the tag needs to be updated.

### Option A: Move v1.0.0 to the fix commit

```bash
git push origin --delete v1.0.0   # remove old tag from origin
git tag -d v1.0.0                  # remove old tag locally
git tag v1.0.0 5506a82              # re-tag at fix commit
git push origin v1.0.0              # tag-push fires workflow
```

**Pros:** v1.0.0 stays as the "first stable" semantic marker. Linear release history.

**Cons:** Moving a tag is functionally a force-push of a release ref. No public Release artifact ever existed for the old v1.0.0 (the upload step never ran), so there's nothing for users to have pulled — but the audit trail of "v1.0.0 was at commit X, then moved to commit Y" lives in the reflog. The auto-mode classifier blocked Claude's first attempt to do this for exactly this reason.

### Option B: Bump to v1.0.1 (Claude's recommendation)

```bash
# Edit apps/configurator/package.json: version "1.0.0" → "1.0.1"
git commit -am "chore: bump configurator version to 1.0.1"
git push
git tag v1.0.1 && git push origin v1.0.1
```

**Pros:** Cleaner audit trail — no tag ever moves. v1.0.0 becomes a "tag-created-but-no-release" historical marker (easy to document or quietly delete from origin later). Standard practice for failed first-tag attempts in OSS projects.

**Cons:** The "first stable" release of TallyMCP Pro is now versioned `v1.0.1` instead of the more aesthetically pleasing `v1.0.0`. Mild semver wart but no functional impact (electron-updater treats them as adjacent versions either way).

### Option C: Delete the broken v1.0.0 tag and stop

```bash
git push origin --delete v1.0.0
git tag -d v1.0.0
```

Leaves main at `5506a82` with the fix in, no tags. Decide later between v1.0.0-reborn or v1.0.1.

## Cursor verdict

**Date:** 2026-05-27  
**Reviewer:** Cursor (implementation review)

### 1. Is `--publish never` the correct fix?

**Yes.** Keep `publish: github` in `apps/configurator/electron-builder.yml`; do **not** switch to `GH_TOKEN` + electron-builder upload for this workflow.

**Why the CI run failed:** In GitHub Actions on a tag ref, electron-builder v25 **implicitly** sets publish policy to `onTag` when no `--publish` flag is passed ([publish — deprecation notice](https://www.electron.build/publish)). With `publish: github` in config, `isPublish` becomes true, the GitHub publisher is created, and upload fails without `GH_TOKEN` / `GITHUB_TOKEN` — matching Run #26491457966 (`artifactCreatedWithoutExplicitPublishConfig` → publisher upload).

**What `--publish never` does:** CLI table: `never` = “never publish” ([CLI / Publishing](https://www.electron.build/cli)) — i.e. skip **upload**, not skip the build. In `PublishManager` (`electron-builder` v25, `app-builder-lib/src/publish/PublishManager.ts`):

- `this.isPublish = … && publishOptions.publish !== "never"` — upload/`scheduleUpload` gated on `isPublish`.
- `createUpdateInfoTasks` / `writeUpdateInfoFiles` run from `artifactCreatedWithoutExplicitPublishConfig` **without** an `isPublish` guard (lines ~223–231, ~260) whenever `publish:` config resolves and the NSIS artifact is suitable.

So with `publish: github` still in YAML, `latest.yml` should land under `apps/configurator/dist-installer/` for `softprops/action-gh-release@v2` to upload. That matches the design in Task 8+9 and the maintainer note on [electron-builder#925](https://github.com/electron-userland/electron-builder/issues/925#issuecomment-261732378) (“file should be generated regardless of publish state”).

**Why not `GH_TOKEN` on the build step:** That re-enables electron-builder’s GitHub publisher alongside `softprops/action-gh-release@v2` — duplicate release ownership, harder to reason about failures, and contrary to the split “build/sign here, upload there” procedure in `docs/v1-installer-phase4-release-procedure.md`. Idempotent double-upload is *possible* but the wrong lever.

**Residual risk (low):** Confirm on the next green CI run that `dist-installer/latest.yml` exists before the upload step (Run #26491457966 may have produced it before the publish error, but the workflow never reached upload). Optional hardening: add a `Test-Path` / `ls` assert on `latest.yml` after `pnpm package` in `release.yml`.

**Also update:** `electron-builder.yml` L66–67 still says CI defaults to `--publish never` outside a tag context — tag-push CI is exactly when implicit `onTag` fired; the root `package.json` explicit flag is what fixes it.

### 2. Tag-move (A) vs v1.0.1 (B) vs delete-and-decide (C)?

**Recommend Option B — bump to `v1.0.1` and delete the orphan `v1.0.0` tag on origin.**

| Option | Fit for this repo |
|--------|-------------------|
| **A — move `v1.0.0` to `5506a82`** | Technically safe (no Release, no artifact). But **rewrites a release ref** already pushed; audit trail shows `v1.0.0` at `db60fc7` then force-moved — awkward for a CA-facing product that cares about traceability. |
| **B — `v1.0.1`** | **Best fit:** immutable tags, linear history, failed `v1.0.0` documented as “tag pushed, CI failed, no release” (note in release issue / CHANGELOG). Matches `docs/v1-installer-phase4-release-procedure.md` §2–3 (version bump → tag → push). Minor semver wart only. |
| **C — delete tag, decide later** | Unnecessary pause; main already has the fix. |

**Concrete sequence (B):**

1. `git push origin --delete v1.0.0` (optional but avoids a dangling tag on the bad commit).
2. Bump `apps/configurator/package.json` `1.0.0` → `1.0.1`, commit on `main` (includes `5506a82` fix).
3. `git tag v1.0.1 && git push origin v1.0.1` — workflow runs on a commit with `--publish never`.
4. After green run: verify four artifacts on the `v1.0.1` release page per procedure §5.

**Do not** use Option A unless you strongly want the marketing label “v1.0.0” and accept documenting a one-time tag force-push in the release notes.
